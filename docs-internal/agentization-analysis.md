# Ultreia Agent 化分析报告

> 内部文档，不面向最终用户。基于 2026-06-19 当前代码和使用手册整理。

## 1. 结论

Ultreia 现在已经不是普通聊天机器人，也不是纯记录工具；它更准确的状态是：

**一个上下文很强、能半自动落地建议的 AI Coach Copilot。**

它离“真正的 agent”还差一层关键能力：**持续目标驱动的自主行动循环**。

现在的 AI Coach 能看很多数据、能给建议、能把建议解析成计划、能在用户确认后写入日历，也能每天主动推送一句提醒。但它本质上仍是“用户问一句，它答一句；用户点按钮，它做一步”。真正 agent 应该能围绕一个目标持续工作：观察状态、发现问题、提出行动、调用工具、执行或等待确认、追踪结果、下次继续纠偏。

所以不要把下一步理解成“让回答更聪明”。更核心的是把 Ultreia 从 **Chat + Context** 推到 **Goal + Tools + Loop**。

## 2. 判断标准：什么才叫真正的 Agent

一个真正可用的 agent，至少有 7 个组成部分：

| 能力 | 说明 | 当前状态 |
|---|---|---|
| 目标 | 用户给它一个持续目标，而不是每次只问一个问题 | 部分具备：目标赛事、计划、Memory 都存在，但还没有“目标任务”对象 |
| 感知 | 能主动读取当前状态、历史、环境变化 | 较强：每次对话注入训练、天气、计划、赛事、Memory |
| 记忆 | 能保留长期事实，并在合适时更新 | 中等：有 Memory，可审核；但不是事实表，也没有时间戳/来源 |
| 规划 | 能把目标拆成步骤、生成计划 | 部分具备：Coach 能给训练建议，能解析计划 |
| 工具 | 能调用明确的动作接口写入数据或触发任务 | 较弱：只有“导入日历”这类 UI 触发动作，没有统一工具层 |
| 执行循环 | 能执行后观察结果，下一次继续修正 | 较弱：计划依从会被注入 prompt，但没有 agent 自己的任务循环 |
| 安全边界 | 动作有权限、确认、审计、回滚 | 部分具备：导入前有 review modal；但没有统一 action log / approval system |

当前最强的是“感知”和“上下文工程”；最弱的是“工具层”和“执行循环”。

## 3. 当前已经接近 Agent 的部分

### 3.1 上下文注入已经很强

AI Coach 每次发消息前会重新拉最新训练记录，并把这些信息拼进 prompt：

- 个人资料和心率区间
- Coach Config
- 长期 Memory
- 当前天气和 7 天预报
- 目标赛事、历史赛事、比赛日天气
- 最近训练、RPE、备注、训练天气
- 8 周训练量趋势
- sRPE / ACWR 训练负荷
- 晨间状态
- 计划依从
- 未来计划
- 当前触发的教练重点

这已经超过大多数“套壳聊天”的水平。Ultreia 的 AI 不是裸聊，它是在读一份结构化训练档案。

### 3.2 已有半自动动作

现在不是只会说建议，已经有几类“说完能落地”的动作：

- AI 回复后可以点 **Import to Calendar**，把建议解析成计划；
- 导入训练后可以直接发送给 Coach 点评；
- Memory 可以由 AI 自动整理，用户审核后写入；
- 每日推送会后台生成提醒，写入 inbox 并推送手机；
- 计划完成情况会自动和实际训练对账。

这些是 agent 化的雏形。

### 3.3 已有最基本的安全设计

目前比较正确的一点是：**AI 不直接偷偷写数据**。

比如导入日历前会弹 review modal；Memory 更新也要 Accept / Discard。这个方向应该保留。训练产品涉及健康建议，不能一上来就让 AI 自主改计划、删计划、加训练量。

## 4. 现在不像 Agent 的地方

### 4.1 没有“任务对象”

现在只有聊天消息、训练记录、计划、赛事、Memory，但没有一个明确的对象叫：

- “备战莫干山 99km 的 agent 任务”
- “这周训练计划监控任务”
- “伤病风险观察任务”
- “天气影响评估任务”

所以 AI 每次都只能在当前对话里临时推理。它没有一个持续任务状态，例如：

```text
目标：2026-10-23 莫干山 99km A 级赛事
当前阶段：基础期 / build 期 / taper 期
本周目标：跑量 55-65km，爬升 1800m+
风险：跟腱、下坡膝盖不适、高温适应不足
下一次检查：周日晚上
待确认动作：下周三改成恢复跑
```

没有任务对象，agent 就没有“连续工作”的抓手。

### 4.2 没有统一工具层

现在 AI 的动作是散落在 UI 流程里的：

- 导入日历是一段专门的 extractor prompt；
- Memory 更新是一段专门的 prompt；
- 每日推送是一段 Edge Function prompt；
- 天气、钱包、训练写入都是各自业务代码。

真正 agent 需要一层明确的工具接口，例如：

```text
create_plan(items)
update_plan(id, patch)
mark_plan_skipped(id, reason)
summarize_week(range)
update_memory_fact(category, content)
fetch_weather(date, location)
create_checkin(date, questions)
```

AI 不应该直接碰数据库，也不应该靠“解析一段自然语言”间接改数据。它应该提出结构化 tool call，然后由应用做权限检查、展示确认、执行、记录结果。

### 4.3 没有行动日志和可追溯性

现在用户能看到聊天历史、钱包流水、inbox，但没有一个“Agent 做过什么”的统一日志。

Agent 化后必须能回答：

- 它刚刚建议了什么？
- 它准备改哪些计划？
- 哪些是用户批准的？
- 哪些执行失败了？
- 为什么失败？
- 失败后有没有重试？

如果没有 action log，agent 一旦开始能写数据，就会很难排查。

### 4.4 主动性还很弱

每日推送是主动的，但它只是“一句提醒”。它不会形成完整流程：

```text
发现：本周计划 60km，但周三已经 ACWR 1.6
判断：继续执行周五 tempo + 周日 long run 风险偏高
行动：提出把周五改成 easy run，周日 long run 缩短 20%
等待确认：用户一键接受 / 修改 / 拒绝
执行：改日历计划
追踪：周日后复盘是否执行
```

这才是 agent。

### 4.5 Memory 仍是“提示词材料”，不是“事实系统”

刚做的分区 Memory 是正确的轻量升级，但它仍是一段文本。真正 agent 需要更细的事实系统：

```text
category: injury
content: 下坡时膝盖容易不适
source: 2026-06-19 chat
confidence: user_confirmed
last_used_at: 2026-06-21
archived_at: null
```

短期不一定要马上建表，但长期看，agent 要判断“哪些事实可信、哪些过期、哪些需要确认”，纯文本不够。

### 4.6 没有评估机制

现在 AI 建议好不好，主要靠用户主观感受。Agent 化后需要最小评估闭环：

- 用户是否接受了建议？
- 导入的计划是否真的完成？
- 完成后 RPE 是否异常高？
- 是否出现漏练、伤病、疲劳？
- 这类建议下次是否应该少给？

没有 eval，agent 只是在“更主动地说话”，不是在变聪明。

## 5. Agent 化的正确方向

### 方向一：先做“可确认的 Action Card”，不要直接自主执行

最适合 Ultreia 的第一步不是“全自动教练”，而是：

**AI 发现问题后生成 Action Card，用户点确认后执行。**

例子：

```text
建议动作：调整本周计划
原因：最近 7 天负荷已经偏高，周五 tempo + 周日 long run 叠加风险过高
将执行：
- 周五 Tempo Run → Easy Run 6km Z2
- 周日 Long Run 24km → 18km
按钮：
[接受调整] [我改一下] [忽略]
```

这比聊天里说一堆建议更 agent，也比直接改日历安全。

可做的 Action Card 类型：

| 类型 | 价值 | 风险 |
|---|---|---|
| 创建训练计划 | 高 | 中，需要确认 |
| 修改未来计划 | 高 | 中，需要确认 |
| 标记计划跳过/完成 | 中 | 低 |
| 更新 Memory | 高 | 低，已有基础 |
| 添加每日状态追问 | 中 | 低 |
| 生成周复盘 | 高 | 低 |
| 发系统推送提醒 | 中 | 中，怕打扰 |

### 方向二：建立一个统一 Agent Action 层

建议新增一个内部抽象，不一定立刻建数据库，先从前端模块开始：

```text
agentActions/
  createPlanAction
  updatePlanAction
  updateMemoryAction
  createReviewAction
  scheduleReminderAction
```

每个 action 都有固定结构：

```json
{
  "type": "update_plans",
  "title": "调整本周计划",
  "reason": "ACWR 偏高，连续高强度风险增加",
  "payload": {},
  "risk": "medium",
  "requiresConfirmation": true
}
```

这样做的好处：

- AI 只负责提出结构化动作；
- 应用负责校验、展示、执行；
- 以后可以把 action 写入数据库；
- 失败/重试/撤销都有位置放。

### 方向三：把“周复盘”做成第一个真正 agent loop

最适合第一个 agent loop 的不是每日推送，而是 **AI 周复盘**。

原因：

- 每日推送太碎，容易打扰；
- 周复盘天然有观察周期；
- 训练计划通常按周安排；
- 成本可控，每周一次；
- 输出可以包含 Action Card。

建议形态：

每周日晚上或周一早上自动生成：

```text
本周复盘
- 实际完成：跑步 48km / 爬升 1200m / 4 次
- 计划偏差：漏掉周三 tempo，周日 long run 缩短
- 风险：ACWR 1.42，接近偏高
- 下周建议：维持跑量，不加强度

待确认动作
1. 下周三改为 Easy Run 8km
2. 周六保留 Long Run，但上限 18km
```

用户点确认后写入日历。

这一步会让 Ultreia 从“聊天工具”明显变成“会主动管理训练周的助手”。

### 方向四：Memory 从“文本”逐步变成“事实”

不要马上上复杂向量库。推荐三阶段：

1. **当前阶段：分区文本**
   - 已做。
   - 成本低，用户可读。

2. **中间阶段：事实卡片 UI，但仍存文本或 JSON**
   - 每条事实可以单独删除/编辑。
   - 有类别、来源、最后确认时间。

3. **最终阶段：Supabase 表**
   - `coach_memory_facts`
   - 字段：`category`, `content_en`, `content_zh`, `source`, `status`, `created_at`, `updated_at`, `last_used_at`
   - prompt 按场景挑选相关 facts。

真正 agent 需要知道哪些事实长期有效，哪些该过期。

### 方向五：把“观察结果”纳入学习

Agent 不能只会建议，还要看建议后来怎么样。

可以先做几个简单信号：

| 信号 | 含义 |
|---|---|
| 用户接受了 Action Card | 建议方向可能有效 |
| 用户修改后再接受 | 原建议部分有效，但细节不准 |
| 用户拒绝 | 建议可能不符合偏好 |
| 计划完成但 RPE 高 | 计划可能偏激进 |
| 计划漏掉 | 可能安排不现实 |
| 漏练后用户备注“太累/出差/疼” | 下次要调整约束 |

这些不用立刻训练模型，只要写进 prompt 和 Memory，就已经能改善下一次建议。

## 6. 推荐路线图

### Phase 1：Action Card 雏形

目标：让 AI 的建议第一次以“可执行动作”形式出现。

范围：

- 复用现有 Import to Calendar，但把它升级成通用 Action Card；
- 第一批只支持：
  - 创建计划；
  - 修改未来计划；
  - 更新 Memory；
- 全部都必须用户确认；
- 先存在前端 state，不急着建 action 表。

验证标准：

- AI 回复后能出现“建议动作”卡片；
- 用户能一键接受；
- 执行结果写入日历或 Memory；
- 失败时有明确错误提示。

### Phase 2：AI 周复盘

目标：建立第一个稳定的 agent loop。

范围：

- 每周自动读取本周训练、计划依从、负荷、下周计划；
- 生成周复盘；
- 写入 inbox；
- 如果有明确建议，附带 Action Card；
- 用户打开后确认是否应用。

验证标准：

- 不需要用户主动提问也能生成有用复盘；
- 复盘能指出计划偏差和负荷风险；
- 至少能生成下周计划调整建议。

### Phase 3：Agent Action Log

目标：让 agent 行为可追溯。

建议建表：

```sql
agent_actions (
  id uuid primary key,
  user_id uuid not null,
  type text not null,
  status text not null,
  title text,
  reason text,
  payload jsonb,
  result jsonb,
  error text,
  created_at timestamptz default now(),
  decided_at timestamptz
)
```

状态：

```text
proposed -> accepted -> executed
proposed -> rejected
accepted -> failed
```

这一步需要 Supabase schema 变更，应该在 Phase 1 验证 Action Card 真的好用之后再做。

### Phase 4：结构化 Memory Facts

目标：让 agent 有长期事实系统。

范围：

- 每条事实有类别、来源、确认状态；
- Memory 审核从“勾选行”变成“事实卡片”；
- prompt 只取相关 facts；
- 旧事实可以归档，不直接删除。

这一步比 Action Card 更重，不建议排在前面。

### Phase 5：外部数据自动同步

目标：减少用户手动上传，增强 agent 的感知能力。

当前结论：

- 短期不把 Strava API 自动同步作为 Ultreia Agent 化路线。技术链路成立（Garmin / 高驰可先同步到 Strava，Ultreia 再接 Webhook / API），但 Strava API 政策限制把 API 数据用于 AI application、RAG、embedding、context window / working memory，以及未经授权的 agent / MCP 场景；这与 Ultreia AI Coach 的核心用法冲突。
- Strava 数据也不等于手表原始 FIT：它能提供摘要和部分 streams，但无法稳定覆盖 Training Effect、RPE、完整原始记录等；心率区间只能在拿得到心率 stream 时近似重算。
- Garmin / 高驰直连如果仍需要企业资质，短期也不碰。
- 日历系统同步可后置。

没有自动同步，agent 的主动性确实会被数据新鲜度限制。但在没有干净、合规、数据质量足够的入口前，不应该为了“自动”牺牲 AI Coach 的政策安全和训练判断质量。短期继续把 FIT / ZIP 作为高质量数据入口，Agent 化优先从 Ultreia 内部闭环推进。

## 7. 不建议做的方向

### 7.1 不要先做“全自动改计划”

训练建议涉及身体风险。直接让 AI 自动修改日历，会让用户失去控制感，也容易出错。正确顺序是：

```text
建议 -> 解释原因 -> 用户确认 -> 执行 -> 记录结果
```

### 7.2 不要先做复杂多 agent

比如“天气 agent / 训练 agent / 比赛 agent / 营养 agent”这种拆分现在没必要。当前规模下，一个 Coach agent 加几个工具就够了。

### 7.3 不要上向量库

Memory 规模很小，训练事实结构也明确。先用分类事实表即可。向量库会增加复杂度，但收益不明显。

### 7.4 不要把 Agent 做成黑箱

Ultreia 的优势之一是用户能看见 Coach 为什么这么说：训练负荷、天气、计划依从都能解释。Agent 化后也要保持这个风格。每个动作都应该有：

```text
为什么建议
准备改什么
会影响哪些数据
是否需要确认
```

## 8. 近期最值得做的 5 个具体功能

按投入产出排序：

1. **Coach Action Card**
   - 把“导入日历”泛化成“建议动作卡片”。
   - 先支持创建/修改计划。

2. **周复盘 inbox**
   - 每周自动生成总结和下周建议。
   - 先不自动改计划，只给确认按钮。

3. **计划调整建议**
   - 当 ACWR 偏高、连续漏练、比赛临近时，自动提出调整未来计划。

4. **Memory 事实卡片化**
   - 把当前分区文本升级成一条条可编辑事实。
   - 暂时可以不建表，先用 JSON 或文本解析。

5. **Agent 行为日志**
   - 记录每次建议、确认、执行、失败。
   - 一旦开始做自动动作，这个必须补上。

## 9. 最小可行 Agent 版本定义

我建议把 Ultreia 的第一个“真正 agent 版本”定义得非常具体：

> 每周自动复盘一次训练和计划，识别一个最重要的问题，提出最多 3 个可确认动作，用户确认后写入日历或 Memory，并在下周复盘时检查执行结果。

这个定义足够小，但已经包含 agent 的核心：

- 主动观察；
- 判断优先级；
- 生成动作；
- 等待确认；
- 执行工具；
- 下次追踪结果。

如果这个闭环跑顺，再考虑更复杂的自主能力。

## 10. 对当前产品定位的影响

Agent 化以后，Ultreia 的定位会从：

```text
AI 私人教练聊天 + 训练记录
```

变成：

```text
会主动管理训练周的 AI Coach Agent
```

这比“回答更聪明”更有产品价值。因为用户真正需要的不是更多文字，而是：

- 什么时候该加；
- 什么时候该收；
- 今天做什么；
- 计划偏了怎么办；
- 下一周怎么改；
- 为什么这么改；
- 改完有没有真的执行。

Ultreia 现在已经有足够的数据基础，下一步应该把 AI 的输出从“建议文本”升级成“可确认动作”。
