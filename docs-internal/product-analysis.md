# Ultreia 产品分析报告

> 内部文档，不面向最终用户。基于 v0.9.4 代码库（2026-06-10）完整阅读后撰写。
> 竞品部分基于 AI 知识库（截至 2026 年初），凡可能过时处均标注 [需核实]。

---

## 0. 产品现状速写（后续章节的事实基础）

- **诞生 28 天**：2026-05-13 从一个 Claude.ai artifact 起步，至今约 4 周，迭代到 v0.9.4，changelog 几乎每天一组更新。
- **技术形态**：Vite + React 单页应用（无 TypeScript、无测试框架、无路由），Supabase 做数据层（10 张表 + RLS + 6 个 Edge Functions），Capacitor 打安卓 APK，PWA 覆盖 Web 端。
- **功能密度远超"玩具"预期**：8 种活动类型、FIT/CSV/ZIP 批量导入（含心率区间逐秒数据 + GPS 轨迹降采样）、个性化 Karvonen 心率区间、sRPE/ACWR 训练负荷、日历计划 + 依从性自动判定、赛事管理 + PR 聚合 + ITRA 指数、彩云天气全链路（实时/预报/历史/比赛日气候常态/超马全程采样）、AI 教练（流式输出、长期记忆、计划一键导入日历）、每日 AI 推送（pg_cron + FCM）、分享海报生成器（5 种类型 × 日夜主题 × 3 种比例，单次海报画真实 GPS 轨迹）、中英双语、应用内 APK 自更新。
- **明显的"个人项目"痕迹**：邀请制注册（`ADMIN_EMAIL` 硬编码为开发者邮箱，constants.js:98）；`App.jsx` 92KB、`AICoachTab.jsx` 85KB 的巨型文件；数据只在启动时拉一次、靠"双击 tab"手动同步；零自动化测试；rolldown tree-shaking 因 bug 整体关闭（bundle 多 ~400KB）。

## 1. 产品定位

**一句话**：给有目标赛事的业余耐力跑者的"AI 私人教练操作台"——记录是喂数据的手段，对话式教练 + 计划闭环才是主菜。

这不是我的概括，是产品自己的声明（docs/README.md）："它不是又一个跑步打卡 app……记录是给教练喂数据的副产品，不是主菜。"代码结构也印证了这一点：全项目最重的两块逻辑是 `coachPrompt.js`（600+ 行的 prompt 数据组装）和 `AICoachTab.jsx`（85KB），而不是记录或图表。

**解决什么问题**：业余跑者备赛时的"没人帮我看数据、没人帮我排课、出了状况没人商量"。具体拆开是三个子问题：

1. **数据散落、没人解读**——手表 app 记了一堆数字但不告诉你"这周加得太猛了"。Ultreia 用 ACWR（`trainingLoad.js`）、8 周量趋势、心率区间分布把原始数据变成风险信号。
2. **请不起/不想请人类教练**——AI Coach 每次对话自动带上 17 类上下文（见第 4 章清单），单条 DeepSeek 消息成本约 $0.002。
3. **建议落不了地**——竞品的 AI 功能大多止步于"分析一下"，这里有"对话 → 一键导入日历 → 自动判定完成/漏掉 → 漏了教练追问原因"的完整闭环（`parsePlansFromLLM` + `evaluatePlanOutcome` + `buildPlanAdherence`）。

**记录工具 vs 教练工具光谱**：明确站在教练侧，大约 70/30。记录功能完整但定位是输入层——证据是产品在记录链路上做的所有增强（RPE 字段、备注、晨间自评、当日标记、天气快照）最终目的都是写进 coach prompt，docs 里每个这类功能的说明都以"喂给 AI 教练"收尾。但它还没到纯教练工具（如 TrainingPeaks 教练版）的程度：已有简化 Fitness / Fatigue / Form（CTL / ATL / TSB 风格）趋势，但没有完整周期化计划模板；主动性已从每日推送扩展到服务端周报和夜间记忆审核，但训练计划动作仍以用户确认后的 Action Card 为边界。

## 2. 目标人群

### 核心：有 A 级目标赛、中高训练量、愿意折腾的中国业余跑者（覆盖度：高）

画像：年跑量 1000km+，备战马拉松/越野/Spartan/Hyrox，用佳明或高驰手表，有过伤病或怕伤病，知道"轻松跑要真轻松"这类概念但执行不稳。

功能覆盖证据：
- 赛事类别直接内置 Spartan（带官方 tier 配色）和 Hyrox（官方组别），这是中国一二线城市新兴的小众赛事人群画像，主流竞品都不单列；
- 备赛全链路覆盖：A/B/C 优先级 → 倒计时 → 周期化阶段提示（`buildFocusDirectives` 按距赛周数输出 Base/Build/Peak/Taper 指令）→ 比赛日天气（两周内真实预报，更早给多年气候常态）→ 热适应提醒（体感 ≥28°C 或湿度 ≥70% 触发）→ 比赛日一键录成绩转历史；
- 中国本土化做到了位：彩云天气、DeepSeek、WGS84 坐标说明里专门提醒"百度地图自带偏移不能直接用"、文档假设用户被 GFW 影响（Claude 走第三方中转 + 香港/日本/新加坡线路）。

**未覆盖的硬缺口**：手表数据不能自动同步——每次都要手动从 Garmin Connect / 高驰 app 导出文件再上传。对这群每周训练 5 次以上的人，这是持续摩擦；但 Strava API 不是干净解法：虽然 Garmin / 高驰 → Strava → API 的技术链路成立，Strava 的 API 政策限制把数据用于 AI application / agent 上下文，且同步数据不等同 FIT 原始记录。短期应承认这个摩擦，而不是用一个会卡住 AI Coach 合规和数据质量的自动同步方案硬补。

### 次级：想脱离教练自己摸索、但不想完全裸跑的进阶跑者（覆盖度：中）

这群人需要的"第二意见"功能是有的（教练人格三轴可调、Prompt 结构可预览），但有两个门槛：10 次免费额度后必须自己申请 DeepSeek key（要注册、充值、建密钥——产品自己都意识到这道坎，内置了分步申请教程），以及数据冷启动——ACWR 要 4 周数据才准（`computeTrainingLoad` 的 `building` 判定），教练前一个月给的负荷建议是降级的。

### 不适合的人群（产品自己也明确排除了）

- **纯记录控/晒图党**：没有社交、没有 feed、没有点赞。海报生成器是给他们的唯一甜头，但导出后去别的平台发。
- **纯小白**：不会问对的问题，也跨不过 API key 门槛。onboarding tour 解决"功能在哪"，解决不了"我该问教练什么"（PromptCatalog 提供了问题模板，算部分缓解）。
- **精英/有人类教练的**：AI 教练的 prompt 是建议式人格（"Do not issue commands"），不做高级周期化编排，满足不了。
- **iOS 用户**：只有 PWA，无推送、Safari 切后台会断请求（docs/ai-coach.md 自己承认）。在中国 iPhone 占有率约两成以上的现实下，这是被动排除的一大群体。

## 4. 产品护城河（诚实评估）

### 4.1 真护城河（有积累效应、难抄）

**用户数据的纵向积累 + 喂给 AI 的"上下文工程"，二者绑定。** 这是唯一够格叫护城河的东西。`buildDataBlock`（coachPrompt.js:419）每条消息组装 17 类上下文：当前时间、实时天气、7 天预报、全部目标赛事、最近一场比赛日天气、精选历史赛事（每类别"最近一场+PR"两个锚点，`selectHistoryForPrompt`）、8 周量趋势、ACWR 负荷、晨间自评、近 10 条训练（含 RPE/备注/逐区间心率/当时天气/超马全程温度曲线）、21 天当日标记、14 天计划依从、未来 21 天计划（7 天内附预报）、条件触发的教练重点。**用户用得越久，这个上下文越值钱，迁移成本越高**——这是经典的数据飞轮，只是飞轮目前只有几个人在转。

**对"为什么不直接问 ChatGPT"的差异化分析**（这是产品存亡问题，值得展开）：

| | 通用 ChatGPT/Claude | Ultreia AI Coach |
|---|---|---|
| 数据输入 | 用户手动复述："我这周跑了 60km……" | 自动注入，且是用户自己懒得复述的维度（逐区间心率、ACWR 数值、计划漏练记录、训练当时的体感温度） |
| 记忆 | 各家 memory 功能不结构化、不可控 [需核实] | 用户可审核可编辑的 500 字事实层 + 逐条 Accept/Discard 的 AI 建议更新 |
| 一致性 | 每次会话人格漂移 | 固定教练人格 + 三轴配置，且 `FIXED_SYSTEM_PROMPT` 写死了反幻觉指令（"不准说看不到数据"） |
| 落地 | 建议停在聊天框 | 解析成结构化计划写进日历，之后自动对账 |
| 主动性 | 纯被动 | 每日定时推送打卡 + 条件触发的教练重点（负荷骤增/临赛减量/漏练追问） |

诚实的另一面：**这个差异化的护城河深度取决于通用助手的进化速度**。ChatGPT/Claude 如果接上 Garmin MCP 或健康数据连接器，"自动注入训练上下文"这层就被釜底抽薪，剩下的只有日历闭环和垂直 UX。这不是远虑——以 2026 年 agent 生态的速度，是 1–2 年内的现实威胁。

### 4.2 只是"暂时没人做"（不是护城河）

- **小众赛事支持（Spartan tier / Hyrox 组别 / ITRA 指数）**：任何竞品一个迭代就能加上，只是市场太小没人理。它是获客钩子，不是壁垒。
- **中国可用性组合（彩云 + DeepSeek + 中转线路）**：纯粹的配置选择，无积累。本土大厂（Keep/咕咚）做 AI 教练的话，这个优势瞬间归零——他们还有设备同步和用户基数。
- **天气深度集成**：做得确实细（超马全程温度采样、比赛日气候常态、prompt 里的热适应规则），但本质是"调 API + 拼字符串"，无壁垒。
- **迭代速度**：28 天做出这些是 AI 辅助开发的展示品，但"开发者 + Claude"这个组合任何人都能复制，而且速度优势伴随质量负债（见 6c）。

### 4.3 反护城河（主动削弱自己的因素）

- **自带 API key 模式**把最核心功能的体验质量外包给了用户的钱包和动手能力；
- **Claude 线路依赖 claudeapi.com 第三方中转**（constants.js:35 自己注明"不是 Anthropic 官方"），中转商跑路或被封，高质量模式直接瘫痪，产品无任何掌控力；
- **邀请制 + 单管理员**：增长被物理关死，数据飞轮转不起来。

## 5. 收费可行性

### 5.1 可能的模式

| 模式 | 适配度 | 说明 |
|---|---|---|
| **Freemium 订阅**（记录免费，AI 教练包月） | 最合理 | AI 是边际成本项（API 调用），订阅对冲成本天然匹配。现有 coach-proxy + usage_quota 的额度架构（coach-proxy/index.ts 的 402 quota_exceeded 逻辑）就是收费墙的雏形，改造成"付费解锁更多额度"技术上是顺路的 |
| 纯订阅 | 不行 | 记录功能弱于一切免费竞品，没资格对记录收费 |
| 买断 | 不行 | AI 是持续成本，买断制等于给自己埋雷；且 APK 侧载分发无法防盗版 |
| 捐赠/爱发电 | 可行但非"收费" | 见 5.4 |

**定价参考区间**：托管 AI 的成本侧，DeepSeek 单条约 $0.002（constants.js 的 pricing 数据），重度用户月 100 条对话 + 每日推送约 $0.3–0.5/月，毛利空间充足。心理价位侧，对标 Keep 会员（¥10–25/月 [需核实]）和 Strava（约 ¥30+/月等值），**¥15–30/月** 是合理试探区间，超过 ¥30 就进入"不如雇真人教练拼单"的对比区。

### 5.2 收费前必须解决的硬伤清单（按致命程度排序）

1. **自带 API key 必须死**。不能一边收钱一边让用户去 DeepSeek 充值。必须转成全托管（服务端 key + 计量），coach-proxy 架构可扩展，但要补：流式输出走代理（现在免费额度是整段返回）、防滥用、成本监控。
2. **无设备自动同步**。收费用户不会接受每次跑完去 Garmin 网页导出文件。但 Strava API 不能再被视为默认解法：它不适合把活动数据送进 AI Coach / 周报 / Memory / Agent Action 判断，且拿不到 FIT 级原始数据。真正收费前要么找到设备厂商的一方授权入口，要么把“自动同步”降级为不进入 AI 的展示能力，并清楚告知数据降级。
3. **无 iOS**。付费产品砍掉 20%+ 的目标设备是自残。
4. **邀请制注册**。要收费先得能注册。
5. **合规成本（中国特有）**：面向中国用户收费 = 需要 ICP 备案（Vercel 部署的境外站收款会很别扭）、个人开发者收款渠道（App 内购/微信支付都需要主体资质）、PIPL 个人信息合规（健康数据是敏感个人信息）、生成式 AI 服务备案风险 [需核实]。这一条对个人开发者可能比前四条加起来还重。
6. **可靠性责任**：零测试、单人维护、Supabase/Vercel 免费层（无 SLA）。收钱之后"数据丢了""服务挂了"从道歉问题变成退款/法律问题。
7. **第三方中转依赖**（Claude 线路）放在付费产品里是定时炸弹。
8. **无社交/留存机制**：订阅制靠续费，纯工具型产品没有外部钩子时流失率高，AI 教练的"关系感"是唯一留存抓手，目前没有数据验证它够不够。

### 5.3 收费的正面论证

如果以上硬伤解决：这个产品的目标用户（备赛严肃跑者）是运动消费里付费意愿最高的人群之一——他们一年花几千上万在跑鞋、参赛费、装备上，¥20/月买"随时在线、看得到我全部数据的教练"在价值感知上是成立的。真人教练市场价 ¥500–2000/月 [需核实]，AI 版收 1/30 的价格做 60% 的事，性价比叙事是通的。且成本结构健康：LLM 推理价格长期下行，毛利会越来越好。

### 5.4 不收费、继续做个人项目的合理性论证

同样成立，且就当下而言**更**成立：

- **成本曲线允许**：自带 key 模式下，开发者的边际成本几乎为零（Supabase/Vercel/GitHub Actions 全免费层，免费额度补贴 = 每新用户 10 条 DeepSeek + 30 次天气，约 ¥0.2）。用户 100 人以内不需要任何收入。
- **收费的固定成本远超收入**：按朋友圈规模（10–50 用户 × ¥20/月 = 月入 ¥200–1000），覆盖不了 iOS 开发者账号（$99/年）+ 合规 + 客服时间 + 可靠性投入。**收费在这个规模是负 ROI。**
- **不收费保留了最大的产品自由**：可以继续邀请制、继续零测试快速迭代、继续用第三方中转，这些在付费语境下全是违约风险，在免费语境下全是合理取舍。
- **它正在履行的真实职能**：开发者自己的训练工具 + AI 辅助开发的学习载体 + 简历级作品。这三个价值不需要收入来兑现。

**我的判断**：当前形态下不收费是对的。收费决策的触发条件应该是"非朋友的陌生人主动来要邀请码的频率"——出现自然需求拉力之前，5.2 的清单不值得做。

## 6. 优化方向

> 额度背景：Supabase 免费层 500MB 数据库 + Edge Functions 50 万次/月调用 [需核实]；Vercel 免费层 100GB 带宽/月；GitHub Actions 公开仓库免费。当前几个用户的负载下，全部余量巨大。

### 6a. 低成本高回报（1–2 周内能做完）

1. **已完成：数据导出/备份（JSON 导出）**。已在删除账号前提供本地 JSON 备份入口；当前仍不是一键恢复，但能覆盖误删或账号异常时的人工查看需求。
2. **已完成：体能趋势曲线（CTL/ATL 简化版）**。Charts 已接入近 56 天 Fitness / Fatigue / Form 趋势，基于 sRPE 日负荷做简化平滑；不等同 TrainingPeaks 官方算法，但已经不是空白。
3. **已完成：AI 周报**。周报已从手动页面推进到服务端定时生成，写入 `coach_reports`，并可从报告提炼计划进入 Action Card。
4. **已完成：教练消息成本可见化**。AI Coach 回复下方会显示实际 token 和内部成本；BYO key 口径已退出普通用户 UI。
5. **仍值得做：coach_messages 与 push_inbox 的清理策略**。两张 append-only 表会无界增长（长对话 + 每日推送落库）。给推送收件箱加保留期、对话清空时真删除或归档，是保护 Supabase 免费层容量的低成本维护项。*额度影响：正面，保护 500MB。*

### 6b. 中期投入（新依赖或架构调整）

1. **外部训练自动同步重新评估（短期不做 Strava API）**。Strava 曾是最现实的自动同步候选：Garmin / 高驰可同步到 Strava，Ultreia 再通过 OAuth + Webhook / 轮询拉活动。但现在不建议推进：Strava API 政策限制 AI application、RAG、embedding、context window / working memory 和未经授权的 agent / MCP 使用；而 Ultreia 的核心价值正是把训练数据送进 AI Coach / 周报 / Memory / Action Card。并且 Strava 拿不到完整 FIT 原始数据，Training Effect、RPE 等关键字段会缺失。除非未来政策和数据范围变化，或只做“不进入 AI prompt 的只读展示”，否则这条线不作为产品主线。*额度影响：暂不发生；短期继续优化 FIT / ZIP 高质量导入。*
2. **iOS（Capacitor 复用现有壳）**。代码层大部分复用，但需要 Mac、$99/年开发者账号、推送链路从 FCM 换 APNs、过审（健康类 + 第三方 LLM 的审核风险）。建议放到"有自然增长需求"之后。*额度影响：零（分发走 App Store）。*
3. **长期记忆结构化**。现在的 memory 是一段 ≤500 字自由文本（`parseBilingualMemory`），覆盖式更新，无时间戳、无主题、容量见顶后旧事实被挤掉。升级成带日期和类别的事实条目列表（伤病/偏好/里程碑分栏），prompt 按需选取。不需要向量库，Supabase 一张表即可。*额度影响：可忽略。*
4. **拆分巨型组件 + 引入最小测试**。App.jsx 92KB / AICoachTab 85KB / ActivitiesTab 70KB，纯 AI 辅助开发的产物，目前能跑是因为只有一个开发者 + 一个 AI 在维护。至少给纯函数层（coachPrompt、trainingLoad、planMatch、fit 解析）补 Vitest 单测——这些是改动最频繁又最怕静默回归的地方（prompt 拼装错了用户根本看不出来，教练只是悄悄变笨）。*额度影响：零。*

### 6c. 大坑预警（看起来诱人，建议别碰）

1. **App 内 GPS 跑步记录**（"既然是跑步 app 为什么不能直接记录？"）。这是从"数据消费端"滑向"数据采集端"的陷阱：持续定位的电量优化、息屏保活（国产 ROM 各自为政）、轨迹纠偏、地图展示在中国需要测绘资质或商业地图 SDK（额度收费）。手表记录的数据质量永远比手机好，这条路投入半年也到不了竞品及格线，而你的用户人手一块手表。
2. **社交功能**（feed、关注、点赞、跑团）。冷启动需要的用户密度差 3–4 个数量级，做出来就是空城，空城比没有更劝退。海报导出借用微信/小红书的社交图谱，已经是这个规模下的最优解。
3. **自建模型/微调"跑步专用 LLM"**。上下文工程（现有路线）的效果天花板高于小规模微调，且零边际成本地继承基座模型升级。微调需要的训练数据（高质量教练对话对）根本攒不出来。
4. **多运动平台化**（铁三/滑雪/攀岩全都管）。Cycling/Swimming 以"不分类、不进跑量统计"的克制方式加入（constants.js 的 `ENDURANCE_GROUP_TYPES` 与 `RUN_GROUP_TYPES` 刻意分离）是对的；但若继续扩成全运动平台，每加一种运动 = 一套子类体系 + 分类规则 + 图表 + prompt 适配 + 文档，把"跑者教练"的定位稀释成"什么都记一点的 app"——那是和 Garmin Connect 正面对撞，必败。
5. **过早全托管 API + 收费**。在自然增长信号出现前做 5.2 清单，是用确定的成本（合规、客服、可靠性责任）换不确定的收入。免费额度 + BYO key 虽然糙，但它把成本风险精确地控制在了零附近——这个糙是当前规模下的正确设计。
